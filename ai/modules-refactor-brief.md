# EBM Refactor Brief: Migrating EBT Repos to electric-book-modules

## Overview

Electric Book Modules (EBM) is an npm package (`@electricbookworks/electric-book-modules`) that abstracts shared JavaScript tooling and front-end code from Electric Book Template (EBT) derivative repos. The reference implementation is **core-the-economy-2** (TE2). This brief documents everything needed to migrate any EBT derivative repo.

Each repo gets an `eb-modules` branch for the refactor. Work one repo at a time; check with the user before proceeding to the next.

---

## Architecture: How EBM Works

### The module (`electric-book-modules`)

- **npm package**: `@electricbookworks/electric-book-modules` (v1.3.4)
- **Provides**: `_tools/`, `_webpack/`, and `assets/js/` (canonical front-end scripts)
- **postinstall script** (`install.js`) runs automatically on `npm install` and:
  1. Empties and copies `_tools/` and `_webpack/` from EBM into the consuming project root
  2. Writes a `.gitignore` inside each copied folder (`*` / `!.gitignore`) so they are not tracked
  3. Copies `_tools-custom/` (if it exists) into `_tools/` with overwrite ŌĆö this is the custom override mechanism
  4. Creates root `gulpfile.js`: `module.exports = require('./_tools/gulpfile')`

### The consuming repo pattern (reference: TE2)

**package.json** changes:
- Add `@electricbookworks/electric-book-modules` as a devDependency (pinned to a tag)
- Remove all raw tool dependencies (chalk, cheerio, gulp, prince, puppeteer, webpack, etc.) ŌĆö EBM bundles them
- Keep only project-specific deps (e.g. `anchor-js`, `holmes.js`, `standard`)
- Add scripts:
  - `"postyalc": "node node_modules/@electricbookworks/electric-book-modules/install.js"` (for yalc dev workflow)
  - `"update-modules": "rm -rf node_modules/@electricbookworks/electric-book-modules && rm -f package-lock.json && npm install"` (clean re-install of modules)
  - `"sync-media"` (if the repo uses media sync)

**Build system**: Jekyll/Liquid `bundle.js` ŌåÆ Webpack `main.js`
- Old: `assets/js/bundle.js` is a Liquid template that Jekyll processes ŌĆö each script is `{% include_relative ... %}`
- New: `assets/js/main.js` is a standard ES module entry point using `import` statements
- Webpack config comes from EBM's `_webpack/webpack.config.js`
- Output goes to `assets/js/dist/main.dist.js`

## Focus Area 1: `_tools` ŌåÆ `_tools-custom`

After `npm install`, EBM's `install.js` copies canonical `_tools/` into the project, then overlays `_tools-custom/` on top. So `_tools/` is gitignored and managed by EBM, while `_tools-custom/` is tracked and contains only project-specific overrides.

### Migration procedure for `_tools`

For each file in the target repo's `_tools/`:

1. **If it matches an EBM file exactly** ŌåÆ do nothing; EBM will provide it
2. **If it exists in EBM but is different** ŌåÆ place the custom version in `_tools-custom/` at the same relative path
3. **If it does not exist in EBM** ŌåÆ place it in `_tools-custom/`
4. **Delete the repo's `_tools/` contents** (they'll be regenerated by EBM's postinstall)
5. **Add `_tools/.gitignore`** (or let EBM create it)

### Critical: `helpers.js` divergence

EBM's `run/helpers/helpers.js` is a thin 12-line wrapper that uses `require-all` to load individual files from `lib/`. Pre-modular repos (like CI2) have a monolithic `helpers.js` that inlines all helper functions directly.

**No action needed**: since EBM copies its own `helpers.js` and full `lib/` directory, the old monolithic `helpers.js` is simply replaced. Only add functions to `_tools-custom/run/helpers/lib/` if the repo has helper functions that don't exist in EBM's `lib/`.

### Update/ directory

Pre-modular repos may have `_tools/update/` (with `files.txt` and `update.sh`) for manual cross-project syncing. This is superseded by EBM and can be deleted.

## Focus Area 2: `assets/js` ŌĆö `bundle.js` ŌåÆ `main.js`

### How the old system works

`bundle.js` is a Jekyll/Liquid template with front matter. Jekyll processes `{% include_relative ... %}` directives to concatenate all scripts. Config data (`settings.js`, `config.js`, `metadata.js`) is injected via Liquid variables from `_data/`.

### How the new system works

`main.js` is a standard ES module entry point. Each script is imported:
- From EBM: `import foo from '@electricbookworks/electric-book-modules/assets/js/foo'`
- From custom: `import bar from './custom/bar'`

Webpack (config from EBM's `_webpack/`) bundles everything to `assets/js/dist/main.dist.js`. Five custom plugins handle config injection:
- `ConfigMergePlugin` ŌåÆ `process.env.config` (replaces `config.js`)
- `YamlEnvPlugin` ŌåÆ `process.env.settings` (replaces `settings.js`)
- `WorksDataPlugin` ŌåÆ `process.env.works` (replaces `metadata.js`)
- `BookIndexFilesPlugin` (replaces `index-loader.js`)
- `FilesPlugin` ŌåÆ `process.env.files`

### Vendor libraries

EBM manages vendor libs as npm dependencies. The `vendor/` directory in pre-modular repos (containing `anchor.min.js`, `elasticlunr.min.js`, `holmes.js`, `lodash.core.min.js`, `mark.min.js`, `marked.min.js`, `prince-box-info.js`, `svg-inject.min.js`) is no longer needed.

These exist in EBM's `assets/js/` and can be imported directly. No custom version needed unless the repo's version has diverged significantly.

#### Category B ŌĆö Significantly diverged (import from `./custom/`)

Compare the repo's file against EBM's version ŌĆö if there are functional differences beyond ES module/style conversion, the repo's own version goes into `custom/` after converting it to an ES module with `export default`.

## Migration Steps (per repo)

### 1. Create branch

```
git checkout master
git checkout -b eb-modules
```

### 2. Update `package.json`

- Set proper `name` and `version`
- Add EBM as devDependency: `"@electricbookworks/electric-book-modules": "github:electricbookworks/electric-book-modules#v1.3.4"`
- Remove raw tool dependencies already provided by EBM
- Keep project-specific dependencies (e.g. `anchor-js`, `holmes.js`)
- Add scripts: `postyalc`, `update-modules`

### 3. Update `.gitignore`

- Add entries for files now installed/generated by EBM:
  ```
  yalc.lock
  .yalc

  # Installed from @electricbookworks/electric-book-modules
  _tools
  _webpack
  gulpfile.js
  ```
- `_tools` and `_webpack` are regenerated by EBM on `npm install`, so they must not be tracked.
- `gulpfile.js` is also written by EBM's postinstall script.
- `yalc.lock` and `.yalc` are used during local EBM development with yalc.
- Use TE2's `.gitignore` as the reference.
- Also create `assets/js/dist/.gitignore` to keep the `dist/` folder in the repo but ignore its build output:
  ```
  # Ignore everything in this directory
  *
  # Except this file
  !.gitignore
  ```

### 4. Create `_tools-custom/`

- Identify files in `_tools/` that are custom (not in EBM, or diverged from EBM)
- Move only those files to `_tools-custom/` preserving directory structure
- Delete `_tools/` content (EBM will regenerate it on install)

### 5. Create `assets/js/main.js`

- Use TE2's `main.js` as a guide on how to refactor bundle.js
- Map each `{% include_relative ... %}` from `bundle.js` to the appropriate import:
  - EBM scripts ŌåÆ `import from '@electricbookworks/electric-book-modules/assets/js/...'`
  - Custom scripts ŌåÆ `import from './custom/...'`
- Preserve the same conditional execution logic, translated from Liquid to `process.env.*` checks
- Preserve script ordering (it matters)

### 6. Create `assets/js/custom/`

- **Source: the repo being refactored, NOT another repo like TE2.** Each file in `custom/` must be derived from this repo's own `assets/js/` version.
- For each JS file in the repo's `assets/js/`, compare against EBM's `assets/js/` version:
  - If identical or only ES module/style differences ŌåÆ import from EBM (no custom file needed)
  - If functionally diverged from EBM ŌåÆ convert the REPO'S OWN version to ES module format, place in `custom/`
  - If not in EBM ŌåÆ convert the REPO'S OWN version to ES module format, place in `custom/`
- Conversion to ES module means: wrap logic in `export default function ebFunctionName () { ... }`, remove `'use strict'`, remove direct invocation, replace `typeof defined !== 'undefined' && defined.Prince` with `process.env.output` checks
- Update any internal references between custom files
- TE2's `custom/` can be used as a REFERENCE for the ES module conversion pattern, but the functional logic must come from the target repo

### 7. Move index files

- Create `_indexes/` at the project root
- Move book-index files with exact rename pattern:
  - `assets/js/book-index-web.js` ŌåÆ `_indexes/book-index-web.js`
  - `assets/js/book-index-app.js` ŌåÆ `_indexes/book-index-app.js`
  - `assets/js/book-index-epub.js` ŌåÆ `_indexes/book-index-epub.js`
  - `assets/js/book-index-pdf.js` ŌåÆ `_indexes/book-index-pdf.js`
  - `assets/js/book-index-print-pdf.js` ŌåÆ `_indexes/book-index-print-pdf.js`
- Move search-index files similarly:
  - `assets/js/search-index-web.js` ŌåÆ `_indexes/search-index-web.js`
  - `assets/js/search-index-with-docs-web.js` ŌåÆ `_indexes/search-index-with-docs-web.js`
  - (and any other output-specific variants present)
- Even though the internal JS search is not used, these files may be needed by other tools (e.g. `build-features-list.js`)
- The `BookIndexFilesPlugin` in webpack expects indexes at `_indexes/book-index-${output}`
- Use `git mv` to preserve history: `git mv assets/js/book-index-*.js _indexes/` and `git mv assets/js/search-index-*.js _indexes/`

### 8. Update `_includes/close-body.html`

- Change the script tag from `bundle.js` to the webpack dist output
- Remove the search-engine.js and search-notifications.js script includes (handled by webpack now)
- Before: `<script data-script-name="bundle" src="{{ path-to-root-directory }}assets/js/bundle.js"> </script>`
- After: `<script data-script-name="main" src="{{ path-to-root-directory }}assets/js/dist/main.dist.js"> </script>`

### 9. Update `_data/settings.yml`

- Restructure flat settings to nested objects to match `process.env.settings` property paths used in `main.js` AND in EBM's own JS files (bookmarks, nav, etc.)
- Key changes: `accordion` (flat boolean + sibling keys ŌåÆ nested object with `enabled`, `level`, `auto-close`), `bookmarks` (flat boolean ŌåÆ nested object with `enabled`, `synchronise`, `note-max-length`, `elements: { include, exclude }`)
- Applies to both `web:` and `app:` sections
- Use TE2's `_data/settings.yml` as the reference for correct structure
- **Critical**: also grep EBM source JS (`node_modules/@electricbookworks/electric-book-modules/assets/js/`) for `process.env.settings` and `settings.` to find ALL required paths ŌĆö not just the repo's own code

### 10. Clean up

- Delete `assets/js/bundle.js`
- Delete `assets/js/config.js`, `settings.js`, `metadata.js`
- Delete `assets/js/vendor/`
- Delete `assets/js/index-loader.js`, `search-engine.js`, `bundle-check.js`
- Delete `_tools/update/` (if present)
- Delete `deploy.sh` and `test.sh` from the project root (these are replaced by EBM's `run.js` commands)

### 11. Update config file JS exclusions

- In `_config.yml` and each format config (`_configs/_config.app.yml`, `_config.epub.yml`, `_config.print-pdf.yml`, `_config.screen-pdf.yml`), replace the long list of individual `/assets/js/*.js` excludes with a single directory exclusion:
  ```yaml
  # Included in main.js, so not needed in build
  - /assets/js/custom
  ```
- The old pattern listed every JS file individually (e.g. `- /assets/js/accordion.js`, `- assets/js/vendor/mark.min.js`, etc.) ŌĆö sometimes 30ŌĆō70 lines. This is replaced by one line because all custom JS now lives in `assets/js/custom/` and gets bundled by webpack into `dist/main.dist.js`.
- Also ensure `vendor` and `_webpack` are in the `_config.yml` exclude list (they should already be there from the original template).
- Use TE2's config files as the reference for the correct pattern.

### 12. Install and test

```
npm install
npm run eb -- output
```

## Execution Playbook

This section captures the proven workflow and learnings from completed refactors.

### Step-by-step process

1. **Explore the target repo** ŌĆö Read `package.json`, `assets/js/bundle.js`, and `_tools/` to understand what's tracked and what's custom.

2. **Compare `_tools/` files against EBM** ŌĆö Use a subagent to read files from both EBM and the target repo. Most files will match EBM exactly. Only files that differ or don't exist in EBM go to `_tools-custom/`.

3. **Compare `assets/js/` files against EBM** ŌĆö Use a subagent to do pairwise comparisons. Classify each as IDENTICAL, MINOR DIFF, or DIVERGED. The verdict determines whether to import from EBM or from `./custom/`.

4. **Update `package.json`** ŌĆö Change name, version, repo URLs. Add EBM dep, remove raw deps, add scripts.

5. **Create `_tools-custom/`** ŌĆö Write the custom override files. Use TE2's `_tools-custom/` as the reference for CORE projects (epub transformations and features command are shared).

6. **Create `custom/` from the repo's own files** ŌĆö For each JS file in the repo's `assets/js/`, compare it against EBM's version. If diverged or not in EBM, convert the REPO'S OWN version to ES module format and place in `assets/js/custom/`. Use TE2's `custom/` only as a reference for the ES module conversion pattern (export default, process.env checks), NOT as the source of the actual file content.

7. **Create `main.js`** ŌĆö Use TE2's `main.js` as template. Adjust for repo-specific includes by cross-referencing with the old `bundle.js`.

8. **Remove old files via `git rm`** ŌĆö Use `git ls-files assets/js/ | grep -v mathjax/ | xargs git rm` and `git ls-files _tools/ | xargs git rm`. Also `git rm gulpfile.js`. Then clean untracked files from disk with `rm -rf`.

9. **Stage new files** ŌĆö `git add _tools-custom/ assets/js/custom/ assets/js/main.js package.json`.

10. **Run `npm install`** ŌĆö EBM's postinstall regenerates `_tools/`, `_webpack/`, and `gulpfile.js`.

### Key learnings

- **Shell files are irrelevant** ŌĆö Ignore `.sh` files during analysis. They are legacy and not part of this refactor.

- **Vendor/ and infrastructure files may not be git-tracked** ŌĆö Check with `git ls-files` before trying `git rm`. They may be in `.gitignore` or be build artifacts. Use plain `rm -rf` for untracked files.

- **`git rm` then `git add` detects renames** ŌĆö When files move from `_tools/` to `_tools-custom/` with identical content, git recognises them as renames in the staging area. This is expected and correct.

- **The old `helpers.js` is simply replaced** ŌĆö Pre-modular repos have a monolithic `helpers.js` (~80+ lines). EBM's thin wrapper + `lib/` directory replaces it entirely. No migration of the monolith is needed unless the repo has helper functions absent from EBM's `lib/`.

- **`featuresList.js` wrapper is needed for `_tools-custom/`** ŌĆö The features command calls `helpers.featuresList(argv)`. EBM's `helpers.js` auto-loads from `lib/` via `require-all`, so the consuming repo needs `_tools-custom/run/helpers/lib/featuresList.js` as a wrapper that calls `build-features-list.js`.

- **Each repo's `build-features-list.js` is unique** ŌĆö CI2's version takes `(bookDirectory)` and reads from `_site/assets/js/search-engine.js`. TE2's version takes `(argv)` with language/format support and reads from `_indexes/search-index-${format}`. Always use the repo's own version.

- **`tables.js` may need to be custom** ŌĆö EBM's `tables.js` only wraps and adds accessibility. CORE repos have responsive positioning logic (`ebPositionTable`, `ebPositionAllTables`, resize listener) that EBM lacks. If the repo has this, convert it to an ES module with `export default` and place in `custom/`.

- **MINOR DIFF files don't need custom overrides** ŌĆö Files that differ only in `'use strict'` markers, direct invocation vs `export default`, or `typeof Prince` vs `process.env.output` checks are handled by EBM's versions. The webpack/ES module system handles these patterns.

- **DIVERGED files go to `custom/`** ŌĆö Files with substantive logic differences (different selectors, extra features, different APIs) must be in `custom/`. **Always use the repo's own version as the source**, converted to ES module format. Do NOT copy from TE2 or another repo ŌĆö each repo's diverged files may have different functional logic. TE2's `custom/` can be used as a reference for the ES module conversion pattern only.

- **Search and bookmark systems are obsolete** ŌĆö Pre-modular repos may have elaborate search files (`search-index-*.js`, `elasticlunr-setup.js`, `search-store.js`, etc.) and bookmark sync. All repos adopt EBM's approach: `search-index.js`, `search-results.js`, `search-terms.js` for search; `bookmarks-legacy.js` for bookmarks. Delete all legacy search/bookmark files.

- **`locales.js` and `nav.js` are DIVERGED but don't need custom** ŌĆö These use Jekyll Liquid globals (`var locales = {{ site.data.locales | jsonify }}`) in the old system. EBM replaces them with `process.env.locales` and `process.env.works` via webpack plugins. Import from EBM directly.

- **Index files must move to `_indexes/`** ŌĆö Pre-modular repos keep `book-index-*.js` and `search-index-*.js` in `assets/js/`. The webpack `BookIndexFilesPlugin` expects book indexes at `_indexes/book-index-${output}` in the project root. Search index files should also move to `_indexes/` ŌĆö although the internal search UI is not used, other tools (e.g. `build-features-list.js`) may still read from them.

- **`_includes/close-body.html` must reference the dist file** ŌĆö The old template loads `assets/js/bundle.js`. Update it to load `assets/js/dist/main.dist.js` and remove the separate `search-engine.js` / `search-notifications.js` script includes (search is now bundled by webpack).

- **Script ordering matters in `main.js`** ŌĆö Preserve the same order as `bundle.js`. In particular: `ebSetup()` first in the web/app block, `ebSvgManagement()` before `ebLazyLoad()`, `ebCookieBanner()` and `ebAnchor()` in the web-only block.

- **`process.env.*` replaces Liquid conditionals** ŌĆö `site.output` ŌåÆ `process.env.output`, `site.build` ŌåÆ `process.env.build`, `site.data.settings.*` ŌåÆ `process.env.settings.*`, `site.audience` ŌåÆ `process.env.config.audience`.

- **`_data/settings.yml` must match `process.env.settings` property paths** ŌĆö The old `settings.js` (a Jekyll/Liquid-generated file) defined properties that JS code accessed as globals. Now webpack's `YamlEnvPlugin` injects `_data/settings.yml` directly as `process.env.settings`. Pre-modular repos use flat YAML keys (e.g. `accordion: false`, `accordion-level: h2`, `bookmarks: true`) but `main.js` and `custom/*.js` reference nested paths (e.g. `settings.web.accordion.enabled`, `settings[output].bookmarks.enabled`). Restructure the YAML to match TE2's nested pattern:
  - `accordion: false` + `accordion-level` + `accordion-auto-close` ŌåÆ `accordion: { enabled: false, level: 'h2', auto-close: false }`
  - `bookmarks: true` ŌåÆ `bookmarks: { enabled: true, synchronise: false, note-max-length: 5000, elements: { include: '.content [id]', exclude: '.expandable-box .toggle' } }`
  - This applies to both `web:` and `app:` sections (app may omit `synchronise` and `note-max-length`)

## EBM Settings Audit: All `process.env.settings` references

This table lists every `process.env.settings` / `settings.` reference in EBM's own JS source files (excluding mathjax). The consuming repo's `_data/settings.yml` must provide values for all of these paths.

**Context**: Webpack's `YamlEnvPlugin` loads `_data/settings.yml` via `yaml.load()` without camelCase conversion. YAML keys like `dynamic-indexing` remain kebab-case in the JS object. `process.env.output` is defined separately via `DefinePlugin`, not inside `process.env.settings`.
